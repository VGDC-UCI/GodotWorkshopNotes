Enemies

Before we start creating our enemy, we should think about what we want our enemy to do. First, we want to be able to spawn them. This tells us we need to make our enemy a Prefab. So, we can go ahead and start with that.

Next, we need to know the type of node our enemy will be. We know we want our enemy to interact physically with our Player, which leaves a few options for node types: Area2D, KinematicBody2D, RigidBody2D, and StaticBody2D.

We want our enemy to move, so that rules out StaticBody2D. We could use a KinematicBody2D, but that would require a bit more work than what is necessary. It would also be nice to be able to let the physics engine handle things like falling and jumping, so it looks like RigidBody2D is the best choice for our enemy.

Our enemy scene should look something like this:

  • RigidBody2D
    • CollisionShape2D
    • AnimatedSprite

If you're using a sprite sheet, your sprite branch will look a bit different, but this is generally how it should look.

Before we get into behaviour, let’s look at how we're going to be spawning our enemies into the main scene. Instead of directly putting our Enemy scene into our main scene, we're going to be making an 'enemy spawner'. This spawner will be creating instances of our enemy scene during the game.

For this “Enemy Spawner” node, all it needs to be is a Node2D with a script. The position of the Node2D will be the place where enemies will spawn in when created.

The other information we need is the maximum number of enemies to spawn, and the interval at which to spawn them. We can export these variables in our spawner's script: 

export(int) var max_enemies
export(int) var spawn_interval

You can set these to whatever you want.
We also need to keep track of how many enemies have spawned, and the time since the last spawn: 

var enemies_spawned: int = 0
var time: float = 0

The last thing we need is the actual enemy scene to spawn. But how do you access this scene from a script? We've never accessed anything from the game's resources before.

We will probably get more in-depth into accessing resources and the file system later, but for now there is a handy class called PackedScene we can use to load the enemy scene. Conveniently, this class can be exported to the editor, so we don't have to mess with paths: 

export(PackedScene) var enemy_prefab

Now you can load the prefab from the editor by finding it in the file system.

Now that we have all the information we need to spawn our enemies, let's put it all together: 

extends Node2D

export(PackedScene) var enemy_prefab
export(int) var max_enemies
export(int) var interval

var time: float = 0
var enemies_spawned: int = 0

func _process(delta):
    if time <= 0 and enemies_spawned < max_enemies:
        spawn_enemy()
        time = interval
        enemies_spawned += 1
    time -= delta

func spawn_enemy():
    var enemy_node: RigidBody2D = enemy_prefab.instance()
    add_child(enemy_node)

The most relevant part of this script is in the spawn_enemy() function. As you can see, we call instance() on the enemy_prefab we have. This is the function that actually creates the node, and we store a reference to it into enemy_node. enemy_node will act just like any other node we would get using the $ operator or using get_node(); however, we then call add_child(enemy_node).

Nodes in Godot don't have to be in the "scene tree" to exist. When we instantiate our enemy scene, it exists somewhere, but it isn't active, and we can't interact with it. In order for a node to be active, it has to exist within the "scene tree". The node doesn't necessarily have to be a child of the node that spawned it, but it does need to be a child of something. However, it doesn't necessarily need to be a child of the scene node. More on that in the Singletons section.

Singletons

You can think of “Singletons” as global nodes that exist outside of the current scene. Until now, every node we’ve dealt with has been part of the “Scene Tree”, that is, they always have a parent, or they are the root node. However, not all nodes are a part of this hierarchy.

In fact, if you’ve used the “remote” scene view to view nodes while the game is active, you’ve seen that there is actually a node above the “root” node in the scene. It’s name is “root”, and it’s a viewport. I won’t go into the implications of that now (you can look up the “viewport” node to find out more if you’d like), but basically this just means that there can be nodes outside of the “scene tree”.

The name Singleton implies Singular, meaning there will only ever be one Object of a Singleton class. This makes Singletons perfectly suited for helper classes, game managers, and storing data you don’t want to be deleted when the scene changes.

When you make a singleton, you are technically making a “sister node” to the root of your scene. That means all singletons are children of the “root” viewport, and sister to the root scene node.

One very nice property of Singletons is that they are in global scope for every script. That means that instead of having to get_node() to access the singleton like you would a regular node, you simply access the singleton by name. If a singleton Bar had a method called foo(), you would call it with Bar.foo().

There are many use cases for singletons, but one we might want to use it counting player deaths. You may have already seen somewhere that the way to change the active scene is to call get_tree().change_scene() or get_tree().change_scene_to() depending on the argument you want to pass. A simple way to restart the level is to call this function and give it the current scene. However, we currently don’t have a way to store data between these restarts, as when a scene reloads, everything is reset to its starting value. Enter: the singleton.

To solve this problem, you would create a new script that inherits from node (extends Node). Once you save the script, you add it as a singleton in Project Settings > Autoload. Click on the folder icon next to “Path” and select your singleton script. Then click add. Your script is now a singleton!

To solve our problem, we need the singleton to keep track of one int. Just make a property for the singleton with a default value of 0.

We want to be able to restart the scene, and when it is restarted, increment the value. Sounds like a perfect thing to make a method for. A reasonable name would be RestartLevel(). This method would add 1 to the death count variable, then call get_tree().change_scene_to() on the current level. Depending on the implementation, you might want to have the level be a parameter to the method to make it more general.

Now whenever the player dies, a node can just call MySingleton.RestartLevel() in order to restart the level, and access the death counter with MySingleton.death_count. (Of course, implementation, names, and such will vary).

Audio

In Godot, there are 3 nodes responsible for Audio: AudioStreamPlayer, AudioStreamPlayer2D, and AudioStreamPlayer3D. As you might imagine, they all work relatively similarly.

One AudioStreamPlayer generally corresponds to one sound you want to play. The sound can be repeated or changed during gameplay, but generally it’s one AudioStreamPlayer per sound.

For things like Music, you’ll want to use the AudioStreamPlayer, because you generally don’t care about the position the sound is coming from, or if the player can hear it or not. AudioStreamPlayers have a set volume, and will play at that volume no matter where the player is.

For sound effects in a game, you’ll want to use the AudioSreamPlayer2D or 3D, depending of course on if the game is 2D or 3D. These nodes take into account how far away the “listener”, or the Camera, is to the sound. It has configurable options for how loud the sound is, how far away the player can be to hear it, and so on. I’d recommend looking up the documentation on these nodes and playing around with them to get a feel for how they work.

AudioStreamPlayer2D/3D nodes are typically children of the objects that make the sound, kinda like how in Unity, the object that makes the sound has an AudioPlayer component. From a script, you just call GetNode(“AudioStreamPlayer2D”) from the parent node and it can control it.

Playing sound is very simple. All you have to do is put the AudioStreamPlayer into a variable (let's say my_sound), and call my_sound.play(). You can give it the time (in seconds as a float) in the song you want to start at as an optional parameter my_sound.play(14.5).
